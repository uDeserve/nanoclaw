## Current Architectural Reality

The code currently implements the first safe host-side medical path.

Important interpretation:

- HealthClaw does not yet run as a fully developed container-side medical
  specialist-agent system
- the current strongest path is a host-side deterministic `symptom_triage`
  pipeline
- the runtime now has two entry shapes:
  - `handleMedicalMessage(...)` for passive message handling
  - `handleHealthEvent(...)` for proactive event handling
- the proactive path now has an explicit orchestration split:
  - active case selection
  - planner adapter invocation
  - planner decision recording
- for HealthClaw-routed messages, the host currently handles classification,
  fact extraction, safety assessment, follow-up planning, patient view,
  expert view, and trace persistence directly
- the generic NanoClaw container agent path remains intact for non-HealthClaw
  traffic

This is consistent with the staged strategy in the project documents:

- build a safe and traceable host-side medical runtime first
- expand to richer specialist-agent orchestration later

The runtime now also has an explicit shared case-state layer, which is the
current bridge between per-template extraction and future planner-oriented
multi-turn behavior.

That bridge is now active for both strongest host-side task paths:

- symptom follow-up continuity
- medication follow-up continuity

## Current Strongest Path

The most complete path today is:

- `symptom_triage`
- `medication_consult` (host-side version with local reference hardening)
- `report_interpretation` (first safe host-side text-report path)
- proactive event simulation (first skeleton only)

What it already does well:

- identifies symptom-like medical messages
- extracts structured symptom fields such as duration, severity, age,
  temperature, location, onset, and associated symptoms
- checks deterministic red-flag rules
- decides a conservative disposition
- asks follow-up questions when required details are missing
- continues a recent incomplete symptom case across turns
- continues a recent incomplete medication consult across turns
- interprets pasted report text through a deterministic first report path
- persists a linked medical trace for audit/debugging

## Current Gaps

The following are not complete yet:

- real planner-agent integration
- richer active-case retrieval across multiple household members or longer case history
- persistent health scheduling
- external-trigger HTTP ingress
- real presence/sensor integration
- report multi-turn continuity
- deep `imaging_qa` logic
- external medical knowledge provider integration
- container-side main-agent plus specialist-agent medical orchestration
